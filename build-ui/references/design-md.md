# DESIGN.md — visual-language source-of-truth (the file, not the skill)

Loaded when `probe.design_md` is a path (i.e., the project has a `DESIGN.md` at
its root). This reference governs **how build-ui consumes** that file. The
sibling `/design-md` skill (pinned in `skills-lock.json`) is a *generator* tied
to Stitch — named here for composition but **not deferred to during normal
build flow**. build-ui consumes DESIGN.md; it doesn't generate it.

## How build-ui uses the file

1. **Read it before writing UI code.** The probe gives you the path; open and
   read it in full. It's small (one file, designed to fit in context).
2. **Treat it as the override layer for visual language.** Where it
   disagrees with generic taste or your training-cutoff understanding of
   the stack's defaults, **DESIGN.md wins**. It's the project's
   committed-on-disk decision.
3. **Don't override it with `references/tailwind.md` or generic patterns.**
   References describe *how to author* in a stack; DESIGN.md describes *what
   the project's visual system actually is*. The latter is more specific.
4. **Carry color names + hex through** when DESIGN.md uses semantic names
   ("Deep Muted Teal-Navy `#294056`"). Use the semantic name in code
   (`bg-teal-navy` Tailwind token, `--color-teal-navy` CSS var) so the
   intent travels. Don't strip the rationale down to a raw hex.
5. **Geometry / shape / atmosphere are real constraints.** "Pill-shaped",
   "Whisper-soft diffused shadows", "Airy density" are not flavor text —
   they're the project's call. Match them.

## Two flavors you may encounter

| Flavor | Source | Looks like |
|---|---|---|
| **google-labs `design.md`** | [github.com/google-labs-code/design.md](https://github.com/google-labs-code/design.md) | YAML front-matter design tokens + rationale prose. Token-first. |
| **Stitch DESIGN.md** | Generated by the `design-md` skill from a Stitch project | Natural-language semantic system with hex codes inline + functional-role tags. Prose-first. |

**Both are valid.** Read whichever is present. Don't normalize the project's
chosen flavor — match it.

## When to mention `/design-md` to the user

Only when *all three* hold:

- The probe returned `design_md: null` (no file).
- The user has Stitch designs they want to extract a DESIGN.md *from*.
- The user actually asks for it ("how do I make a DESIGN.md", "extract a
  design system from my Stitch project").

Otherwise, don't surface it — it's a Stitch-only generator, not part of
build-ui's flow.

## Composition stance

`frontend-design` (taste) → `source-ui` (visual precedent) → **DESIGN.md (project's
committed visual language) → build-ui (execution that respects it)**. When
DESIGN.md exists, it slots in *above* generic taste and *below* the project's
own components — it's the bridge.

If DESIGN.md is absent, fall back to the rest of `references/` and the
detected stack defaults. No degradation message needed — most projects
don't have one.
