# Accessibility — not optional

Loaded by `SKILL.md` Step 3 **always** (regardless of stack). a11y is part of
"production-grade," not a separate feature. Most failures come from a few
recurring mistakes — get those right and the rest is mostly fine.

## The non-negotiables (verify before considering work done)

1. **Semantic HTML first.** Use `<button>`, `<a href>`, `<nav>`, `<header>`,
   `<main>`, `<footer>`, `<section>` (with a heading), `<article>`,
   `<label for>`, `<fieldset>/<legend>`. Reach for ARIA only when no element
   says it for you.
2. **One `<h1>` per page**, then `<h2>`/`<h3>` in order without skipping.
   Screen-reader users navigate by heading; a broken hierarchy strands them.
3. **Every interactive element is keyboard-reachable** (`Tab`/`Shift+Tab`) and
   has a **visible focus ring** that isn't hidden by `outline: none` without a
   replacement. `:focus-visible` is the right hook for keyboard-only rings.
4. **Form fields have associated labels.** `<label for="x">` + `<input id="x">`
   (or wrap the input in the label). Placeholder text is NOT a label.
5. **Color contrast ≥ 4.5:1** for body text, ≥ 3:1 for large text and meaningful
   UI (focus rings, borders that carry state). When in doubt, sample with a
   contrast checker.
6. **Respect `prefers-reduced-motion`.** Wrap meaningful animations in
   `@media (prefers-reduced-motion: no-preference)` (note the *positive* check
   is the safer default), or use the JS equivalent.
7. **Images have `alt`.** Decorative: `alt=""` (empty, not missing). Meaningful:
   describe the *function* of the image, not its appearance. Icon buttons need
   an accessible name via `aria-label` or visually-hidden text.

## Anti-patterns to stop on sight

- **`<div onClick>` as a button.** It's not a button — no keyboard, no role,
  no native focus. Use `<button>`.
- **Link-as-button or button-as-link.** Links go places; buttons do things. A
  router push is *navigation*, use `<a>`/`<Link>`. A modal toggle is *action*,
  use `<button>`.
- **Hidden focus rings.** `outline: none` without a replacement is the
  single most common a11y regression. `:focus-visible` ring → fine. Nothing → no.
- **`autofocus` on page load.** Almost always wrong; jumps a screen-reader user
  away from the heading they were about to read.
- **`tabindex` > 0.** It rearranges DOM tab order in unintuitive ways. Use 0
  (default-focusable) or -1 (programmatic focus target) and let the DOM order
  do the work.
- **Toasts and live regions without `aria-live`.** `aria-live="polite"` for
  most updates; `assertive` for urgent (errors). Don't overuse — interruption
  fatigue is real.
- **Color-only state.** A red border on an invalid field needs a non-color cue
  too (icon, message). Same for "in stock" / "out of stock."

## Component-class specifics

- **Modals/Dialogs:** trap focus inside while open; restore focus to the trigger
  on close; close on `Esc`; the overlay is non-interactive content (use
  `inert` on the rest of the page if your runtime supports it). Radix/shadcn
  Dialog gets all of this right — use it instead of hand-rolling.
- **Menus/Dropdowns:** arrow keys move within; `Esc` closes; first item gets
  focus on open; the trigger has `aria-haspopup` / `aria-expanded`.
- **Tabs:** arrow keys to switch; one tabstop for the tab list (not one per
  tab); `aria-selected` on the active tab; panel `role="tabpanel"`.
- **Tooltips:** appear on `:focus-visible` *and* hover; never carry the only
  copy of meaningful information.
- **Forms:** required → `required` attribute (announces); errors → text near
  the field *and* `aria-describedby` linking the input to the error id.

## Verifying

If the user has `automate-browser` or `example-skills:webapp-testing`, hand
the verify step to them — keyboard-only walk through the change, check the
heading outline (e.g. with the browser's accessibility tree), verify
`prefers-reduced-motion` short-circuits animations, sample contrast on the
edited surfaces. If neither is set up, note that visual verification is the
user's call and list the concrete keyboard/contrast/reduced-motion checks.

## Why this is in build-ui (not its own skill)

a11y isn't a stack — it's a *layer on every stack*. Folding it into build-ui
keeps it from being skipped. It's loaded on every run.
