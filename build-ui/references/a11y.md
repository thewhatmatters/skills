# Accessible Component Checklist

Loaded by `SKILL.md` Step 3 **always** (regardless of stack). Use the **Universal
Pre-Flight** for every component you build, plus the section that matches the
specific pattern.

**Operating principle:** semantic HTML first → native elements before ARIA →
ARIA only to fill genuine gaps. Most accessibility failures come from rebuilding
native behavior badly, not from missing ARIA. "No ARIA is better than bad ARIA"
— but *good* ARIA where the platform leaves a gap is the actual goal.

> **Stack note** (matches `references/shadcn.md`): If the project is on React +
> shadcn/ui, the Radix primitives underneath already implement most
> keyboard/focus/ARIA behavior for dialogs, tabs, dropdowns, comboboxes, etc. In
> that case this checklist is mostly for (a) verifying you didn't break those
> guarantees with custom markup, and (b) the content-level concerns primitives
> can't solve: labels, alt text, contrast, copy, error handling, focus
> restoration targets.

---

## Universal Pre-Flight (every component)

- [ ] **Right element for the job.** Navigation → `<a href>`. Action → `<button type="button">`. Never `<div onclick>`. If it does something, it's a button; if it goes somewhere, it's a link.
- [ ] **Keyboard operable.** Everything achievable with a mouse is achievable with a keyboard. Tab to reach, Enter/Space to activate, Escape to dismiss, Arrows to move within a composite widget.
- [ ] **Visible focus indicator.** Never `outline: none` without a replacement. Focus ring has ≥3:1 contrast against adjacent colors and is clearly distinguishable.
- [ ] **Focus is never trapped** (except intentionally inside an open modal, which must offer Escape + a close control).
- [ ] **Logical focus order** matches visual/reading order. No positive `tabindex`. Use `tabindex="0"` (in order) and `tabindex="-1"` (programmatic focus only).
- [ ] **Accessible name** on every interactive element (visible label, `aria-label`, or `aria-labelledby`). Icon-only controls always need one.
- [ ] **State exposed to AT** via correct role + ARIA state (`aria-expanded`, `aria-selected`, `aria-checked`, `aria-disabled`, `aria-current`, etc.).
- [ ] **Color is not the only signal.** Errors, required fields, selected states, links-in-text all carry a non-color cue (text, icon, underline, shape).
- [ ] **Text contrast ≥ 4.5:1** (≥3:1 for large text ≥24px / 19px bold). **Non-text UI contrast ≥ 3:1** (borders, icons, focus rings, control boundaries).
- [ ] **Target size ≥ 24×24 CSS px** (WCAG 2.2 AA), or adequate spacing around smaller targets. Aim for 44×44 for primary touch targets.
- [ ] **Respects `prefers-reduced-motion`** — disable/reduce non-essential animation and parallax.
- [ ] **Works at 200% zoom and 400% reflow** (320px viewport equivalent) with no loss of content or horizontal scrolling of body text.
- [ ] **Decorative imagery hidden** from AT (`alt=""` or `aria-hidden="true"`); meaningful imagery has descriptive `alt`.

## Forms & Inputs

- [ ] Every field has a **programmatically associated `<label>`** (`<label for>` → `id`), or `aria-label` / `aria-labelledby`. Placeholder text is **not** a label.
- [ ] **Required** indicated by the `required` attribute **and** a non-color visual cue; consider `aria-required="true"` for custom controls.
- [ ] **Errors**: set `aria-invalid="true"`, link the error message with `aria-describedby`, describe how to fix it (not just "invalid"). On submit, move focus to the first error (or an error summary) and ensure it's announced.
- [ ] **Help/hint text** linked via `aria-describedby`.
- [ ] **Group related controls**: radio/checkbox groups wrapped in `<fieldset>` + `<legend>`.
- [ ] **Correct input types** (`email`, `tel`, `url`, `number`, `date`) and **`autocomplete`** tokens (WCAG 1.3.5) so user data prefills.
- [ ] Disabled vs. read-only chosen deliberately; avoid `disabled` on elements users need to perceive (it removes them from tab order and lowers contrast).
- [ ] **Accessible authentication** (WCAG 2.2): allow paste, password managers, and copy in auth fields. No cognitive puzzles as the only method.
- [ ] **Redundant entry** (WCAG 2.2): don't force re-typing info already provided in the same flow.

---

## Button

- [ ] Native `<button>`. Set `type="button"` unless it submits a form (`type="submit"`).
- [ ] Activated by **Enter and Space**.
- [ ] Icon-only buttons have an accessible name (`aria-label` or visually-hidden text).
- [ ] Toggle buttons expose `aria-pressed`; controls that show/hide use `aria-expanded` (see Disclosure).
- [ ] Loading/busy state communicated (e.g., `aria-busy`, or swap label and disable re-entry without silently removing focus).

## Link

- [ ] Native `<a>` with a real `href`. Activated by **Enter** (not Space).
- [ ] Link text is meaningful out of context — no bare "click here" / "read more" (or supplement with `aria-label` / visually-hidden context).
- [ ] Links visually distinguishable from body text by more than color alone (underline or ≥3:1 contrast + non-color cue).
- [ ] Opens-in-new-tab is signaled in the accessible name when it happens.

---

## Modal Dialog

Use native `<dialog>` + `.showModal()` where possible; it handles the inert backdrop and Escape for free. Otherwise replicate every line below.

- [ ] `role="dialog"` + `aria-modal="true"` (use `role="alertdialog"` for confirmations requiring a decision).
- [ ] Labeled by its title via `aria-labelledby`; optional `aria-describedby` for body/instructions.
- [ ] **On open: focus moves into the dialog** (first interactive element, or the heading/dialog container).
- [ ] **Focus trapped**: Tab / Shift+Tab cycle only within the dialog while open.
- [ ] **Escape closes** the dialog (unless explicitly an alertdialog requiring a choice — even then, offer an obvious cancel).
- [ ] **On close: focus returns to the triggering element.**
- [ ] **Background is inert** — use the `inert` attribute (or `aria-hidden`) on everything outside the dialog so AT and keyboard can't reach it.
- [ ] Closing affordance is reachable and labeled (e.g., a "Close" button).

| Key | Action |
|---|---|
| Tab / Shift+Tab | Cycle focusable elements inside the dialog (does not leave) |
| Escape | Close, return focus to trigger |
| Enter / Space | Activate the focused control |

## Disclosure (show/hide) & Accordion

- [ ] Trigger is a `<button>` with **`aria-expanded`** (`true`/`false`) and **`aria-controls`** pointing to the region's `id`.
- [ ] Enter/Space toggles; the controlled region is shown/hidden in sync with `aria-expanded`.
- [ ] **Accordion**: each header button is wrapped in an appropriate heading (`<h2>`–`<h6>`) for outline navigation. Don't nest buttons inside buttons.
- [ ] Arrow-key movement between accordion headers is optional (APG allows it); if added, Home/End jump to first/last. Tab must still work normally.

---

## Tabs

- [ ] Container `role="tablist"`; each tab `role="tab"`; each panel `role="tabpanel"`.
- [ ] Selected tab: `aria-selected="true"` (others `false`). Tab → panel via `aria-controls`; panel → tab via `aria-labelledby`.
- [ ] **Roving tabindex**: active tab `tabindex="0"`, inactive tabs `tabindex="-1"`. Tab key enters the tablist on the active tab, then exits to the panel.
- [ ] Arrow keys move **between tabs**; Home/End to first/last.
- [ ] Choose **manual activation** (focus a tab, press Enter/Space to select) when revealing a panel is expensive; **automatic activation** (select on focus) is fine for cheap panels.

| Key | Action |
|---|---|
| Left/Right (or Up/Down if vertical) | Move focus between tabs |
| Home / End | First / last tab |
| Enter / Space | Activate focused tab (manual activation) |
| Tab | Move from tablist into the active panel |

---

## Menu Button & Menu

(For a list of *commands/actions*. For navigation, prefer plain links in a `<nav>` — don't reach for `role="menu"`.)

- [ ] Trigger `<button>` with `aria-haspopup="menu"` and `aria-expanded`.
- [ ] Menu container `role="menu"`; items `role="menuitem"` (or `menuitemcheckbox`/`menuitemradio`).
- [ ] **On open, focus moves to the first item** (or the previously active item).
- [ ] Up/Down arrows navigate; Home/End jump; typing a letter moves to a matching item (optional but expected).
- [ ] **Escape closes and returns focus to the trigger button.**
- [ ] Selecting an item activates it and closes the menu.

---

## Combobox / Autocomplete / Custom Select

Prefer native `<select>` when the options are a simple list. Build a combobox only when you need filtering/typeahead.

- [ ] Input has `role="combobox"`, `aria-expanded`, and `aria-controls` → the listbox `id`.
- [ ] Popup is `role="listbox"`; options are `role="option"` with `aria-selected`.
- [ ] Active option tracked with **`aria-activedescendant`** (focus stays on the input) — don't move DOM focus into the list for the editable pattern.
- [ ] Down/Up arrows move through options; Enter selects; Escape closes (and/or clears) and returns to the input; Home/End supported.
- [ ] Results count / "no results" announced via a live region.

---

## Tooltip & Content on Hover/Focus

- [ ] Trigger references the tip with `aria-describedby`; tip has `role="tooltip"`.
- [ ] Appears on **both hover and keyboard focus** — never hover-only.
- [ ] **Dismissible** (Escape hides it without moving focus), **hoverable** (can mouse over the tip without it vanishing), and **persistent** (stays until dismissed or focus/hover leaves) — WCAG 1.4.13.
- [ ] Don't put interactive content or essential-only information in a tooltip; don't trap focus in it.

---

## Live Regions (toasts, status, alerts)

- [ ] Region exists in the DOM **before** content is injected (AT may miss content added together with the container).
- [ ] `role="status"` / `aria-live="polite"` for non-urgent updates (saved, copied, results loaded); `role="alert"` / `aria-live="assertive"` for urgent/error messages only.
- [ ] **Don't steal focus** for transient toasts. If the toast contains an action (Undo), make sure that action is keyboard reachable and the toast persists long enough (or is dismissible).
- [ ] Don't over-announce: avoid `assertive` for routine updates.

---

## Tables

- [ ] Real `<table>` for tabular data (not divs). `<caption>` describes the table.
- [ ] Header cells use `<th>` with `scope="col"` / `scope="row"`; complex tables use `headers`/`id` association.
- [ ] Don't use tables for layout.

## Cards / Large Clickable Regions

- [ ] Don't attach a click handler to a whole `<div>`. Put a real `<a>`/`<button>` on the primary action; if you want the whole card clickable, use the "stretched link" technique (heading contains the link, `::after` covers the card) so only one focusable target exists.
- [ ] Nested interactive elements (a button inside a clickable card) remain independently reachable — avoid nesting interactive inside interactive.

## Carousels & Drag Interactions

- [ ] **Dragging has a single-pointer alternative** (WCAG 2.2 2.5.7): reorder/resize/slider actions also operable via buttons or taps, not drag-only.
- [ ] Carousel: pause/stop control for auto-advancing content; prev/next are real buttons; current slide state exposed.
- [ ] Custom sliders implement the APG slider keyboard pattern (Arrows adjust, Home/End to bounds, `role="slider"` with `aria-valuenow/min/max/text`) — or use `<input type="range">`.

## WCAG 2.2 deltas to watch (newer criteria, easy to miss)

- [ ] **2.4.11 Focus Not Obscured (AA)** — a focused element is never fully hidden behind sticky headers/footers or non-modal overlays.
- [ ] **2.5.7 Dragging Movements (AA)** — single-pointer alternative to any drag operation (see Carousels).
- [ ] **2.5.8 Target Size (AA)** — interactive targets ≥ 24×24 CSS px or sufficiently spaced.
- [ ] **3.2.6 Consistent Help (A)** — help/contact mechanisms appear in the same relative location across pages.
- [ ] **3.3.7 Redundant Entry (A)** — don't re-ask for info already given in the flow.
- [ ] **3.3.8 Accessible Authentication (AA)** — no cognitive-function test as the only path; allow paste and password managers.
- [ ] (AAA, aspirational) **2.4.13 Focus Appearance**, **2.4.12 Focus Not Obscured (Enhanced)**, **3.3.9 Accessible Authentication (Enhanced)**.

Note: WCAG 2.2 is a superset of 2.1 (it removed the old 4.1.1 Parsing criterion). Meeting 2.2 means you meet 2.1. Semantic markup and correct ARIA still matter even though parsing was dropped.

## Verification protocol (run before shipping)

**1. Automated scan** (catches roughly a third of issues — necessary, never sufficient)
- [ ] `axe-core` (or `@axe-core/playwright`), Lighthouse a11y, or Pa11y in CI. Zero violations on the component's stories/states.
- [ ] Lint with `eslint-plugin-jsx-a11y` (React) for the obvious markup mistakes.

**2. Keyboard-only pass** (unplug the mouse)
- [ ] Reach every control with Tab/Shift+Tab in a logical order.
- [ ] Operate every control with Enter/Space/Arrows/Escape per the pattern above.
- [ ] Focus is always visible and never lost, trapped (outside modals), or sent somewhere surprising.
- [ ] Modals: focus enters on open, traps, Escape closes, focus returns to trigger.

**3. Screen reader pass** (at least one desktop + one mobile)
- [ ] NVDA + Firefox (Windows) and/or VoiceOver + Safari (macOS/iOS); JAWS + Chrome if your audience needs it.
- [ ] Name, role, and state are announced correctly for each control; dynamic updates are announced via live regions; nothing is silent or mislabeled.

**4. Visual / robustness**
- [ ] 200% zoom and 400% reflow (≈320px) — no clipped content or horizontal scroll of text.
- [ ] Color contrast checked on text, icons, borders, focus rings, and each interactive state.
- [ ] Windows High Contrast / forced-colors mode doesn't erase essential UI.
- [ ] `prefers-reduced-motion: reduce` honored.

If the user has `automate-browser` or `example-skills:webapp-testing`, hand the
keyboard + screen-reader walk to them. Otherwise list the concrete
keyboard/contrast/reduced-motion checks the user needs to run.

### TL;DR decision order for any component

1. Can a **native element** do this? Use it.
2. Is the **keyboard interaction** the one users already expect (APG)? Match it exactly.
3. Does **focus** go somewhere sensible on open, move predictably, and return on close?
4. Is every control **named, with role and state** exposed to AT?
5. Does it survive **keyboard-only + screen reader + 200% zoom + contrast** checks?

If all five are yes, the component is accessible. If you're adding ARIA to answer #4, double-check you didn't skip #1.
